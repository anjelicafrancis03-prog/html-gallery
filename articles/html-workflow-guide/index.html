HTML Workflow / Codex 多线程成品规范

HTML 工作流：从图文长文到多平台发布

这是一套给所有线程复用的 HTML 生产线：先做高质量母版，再做平台发布包，所有产物统一落盘、统一验收、统一交接。

固定总仓库：C:\html 母版：articles\<slug>\index.html 验收：localhost + 桌面/手机
总原则 目录规范 Skill 组合 内容结构 图片规则 验收 平台封装 交接
01 / principle

HTML 不是网页草稿，而是内容母版

以后所有专门文章都先做成 HTML 母版。母版负责完整表达：结构、节奏、图文关系、证据截图、视觉层次和交接口令。公众号、飞书、知乎、小红书都不是直接复制母版，而是从母版降级成各自能稳定承受的版本。

一句话：HTML 是母版，localhost 是验收线，publish 是平台版，C:\html 是总仓库。

1个 HTML 母版，保留完整排版。
3+类平台发布包，按能力降级。
4类关键资产：图片、封面、截图、验收记录。
0容忍项：坏图、乱码、横向溢出。
02 / directory

固定落盘：所有成品进入 C:\html

目录固定以后，别的线程不用猜文件在哪里，平台大师也不用到处找图片。每篇文章一个独立子目录，所有素材跟着文章走。

C:\html\articles\<slug>\index.html
C:\html\articles\<slug>\assets\...
C:\html\articles\<slug>\publish\...
C:\html\articles\<slug>\imgs\...

新增成品后必须更新 C:\html\README.md。这不是仪式感，是让其他线程能直接接手的索引。

03 / skill stack

Skill 组合：不是一个工具包打天下

HTML 文章通常不是单一 skill 能做好的。我的默认组合是：用 html-anything 做成品化，用 article-magazine 做长文阅读版式，用 html-visual 做信息图和视觉化模块，缺关键图时用 image2。

html-anything

把复杂材料、报告、目录、网页内容转成可浏览的 HTML 成品。

article-magazine

负责长文版式、阅读节奏、标题层级和文章感。

html-visual

负责对比矩阵、流程图、决策图、数据卡片等视觉化表达。

image2 / gpt-image-2

缺主题图时生成，但必须和内容直接相关。

浏览器验收

用 localhost 和截图检查真实显示，不靠想象过关。

agentmemory

把路径、口令、踩坑经验和工具状态沉淀给未来线程。

04 / structure

内容结构：先给结论，再给证据

测评、工具介绍、技术解释、平台发布经验，默认都遵守一个顺序：先让读者知道结论，再让他看到方法和证据，最后给出可执行的口令。

首屏结论

这东西到底好不好用、适合谁、风险在哪里。

读者判断
背景痛点

为什么要做这篇，解决什么具体问题。

建立语境
测试方法

测试环境、工具版本、样本和复现路径。

可信度
结果对比

成功、失败、边界，用表格或卡片表达。

可扫描
截图证据

真实截图、真实输出、真实文件路径。

不空口
交接口令

给其他 Agent 可以直接复制执行的指令。

可复用
05 / images

图片规则：图要服务内容，不要只负责好看

图文并茂不是配几张气氛图。文章里的每张图都要承载一个明确功能：总览、证据、对比、流程、目录或平台差异。

图片类型用途注意事项
真实截图证明工具、页面、结果真实存在优先使用，不随意裁剪关键区域
信息图压缩流程、目录、平台规则一张图只表达一个结论
封面图公众号/社媒首图和分享卡片单独生成尺寸，不混用正文图
生成图缺关键视觉时补主题图必须贴合主题，不能只当装饰
 
.figure-card img {
  width: 100%;
  height: auto;
  object-fit: contain;
}

历史教训很明确：不要把多张信息图横向并排，不要用 object-fit: cover 裁剪中文信息图，不要把深色信息图硬塞进浅色文章。

06 / validation

验收：file:// 不算完成

带本地图片的 HTML 必须用 localhost 验收。浏览器真实渲染能暴露路径错误、图片坏链、移动端横向滚动、代码块撑破页面和中文乱码。

cd C:\html
python -m http.server 4174 --bind 127.0.0.1

http://127.0.0.1:4174/articles/<slug>/index.html
HTTP 200
页面能通过本地服务打开。
无乱码
没有连续问号乱码，标题和正文中文正常。
图片全加载
所有 naturalWidth 都大于 0。
无横向溢出
桌面和 390px 手机宽度都不能横滚。
关键区截图
首屏、图片区、表格区、移动端都要看。
README 更新
新增文章必须登记到总目录。
07 / platform

平台封装：不是复制 HTML，而是稳定降级

公众号、飞书、知乎、小红书对 HTML 的支持完全不同。母版可以精致，发布版必须稳。真正的目标不是“所有平台看起来完全一样”，而是“每个平台都能舒服阅读，不丢关键证据”。

平台保留什么降级什么
公众号标题、段落、单列图片、封面、核心卡片JS、复杂 grid、表格、SVG、canvas
飞书文档结构、表格、图片、视频附件CSS、Chart.js、网页交互、主题切换
知乎逻辑、论证、图片、引用复杂排版和装饰性布局
小红书/社媒结论和视觉卡片长文结构，改成多张高 DPI 图片
08 / handoff

给其他线程的最短交接口令

其他线程不需要重新理解整套系统，给它们一段固定口令就够了。

HTML 成品统一落盘到 C:\html\articles\<slug>\index.html。
图片放 C:\html\articles\<slug>\assets\。
完成后启动 C:\html 的 4174 本地服务验收。
不要只用 file://。
检查桌面、手机、图片区、表格区。
有平台发布需求时另做 publish 版，不要直接拿 index.html 发平台。

如果是公众号：表格改卡片，图片单列，CSS 内联，复杂交互转 PNG。 如果是飞书：保留结构，不保留 CSS；图表转表格或截图，视频做附件。

本文即为 HTML 工作流母版示例，落盘于 C:\html\articles\html-workflow-guide\index.html。后续平台发布版可从该母版继续生成。
