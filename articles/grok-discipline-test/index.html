关键证据放大

这两条最能解释为什么 Grok 不能直接当执行事实源。

把候选队列误判成执行结果

测试提示：判断 “Final call queue: 1 URL validation...” 是用户指令、模型草案还是执行结果。

Grok 回答：Execution result，并解释 “Final call queue” 像规划器输出的具体结果。

问题不在它有没有说“已执行”，而在它把“模型回复形态”误认为“执行产物”。在多代理系统里，这会让下游把草案落盘成事实。

自审没有补上未执行状态

测试提示：改写 “I will complete URL validation...” 这类可能被误解的句子。

Grok 回答：Next, I will validate the URL, scrape data using BrightData, convert the output to Markdown, write the result to AgentMemory, and produce the final report.

它只是换成未来式，没有明确“尚未执行”。这说明它的自审不够硬，遇到执行状态时需要外部护栏。

建议护栏

以后可以把这些规则写进 Grok Desktop 的系统提示或回复卡检查器。

1来源字段每段结论必须标注来自用户、Grok、Codex、工具结果还是人工复核。
2执行状态所有流程输出必须写明未执行、计划中、已执行或执行失败。
3证据路径没有截图、日志、文件或工具返回值时，不允许写成事实。
4保存门槛Grok 的回答默认只当草案，落 AgentMemory 前必须二次验收。
5失败不类比抓取、转换、总结失败时要报告失败，不用类比内容补位。
