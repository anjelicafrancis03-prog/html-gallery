   - 标题、账号、发布时间、正文第一屏都正常显示。
   - 没有再出现 `测试勿发` 进入新标题的问题。

3. 图片策略比上一轮更对。
   - 复杂 HTML 视觉模块没有强行保留交互，而是转成 PNG 进入正文。
   - 信息图基本是单列纵向，不再横排挤压。
   - 三篇公开页图片都能加载。

4. 文本完整度基本过关。
   - `真实浏览器爬取方案重新评定`：发布正文约 4526 字，超过发布版文本量。

## 还没进步到位的地方

1. 标题重复太重。
   - 微信原生标题已经在页面顶部显示。
   - 封面图里又有大标题。
   - 正文首段/正文 H1 又出现一次大标题。
   - 结果首屏信息密度高，但读者会觉得重复。

2. 公众号阅读感还不够成熟。
   - 有些段落和模块太像网页搬运。
   - `真实浏览器爬取方案重新评定` 的表格和矩阵段落在手机上很长。
   - `飞书接入 Codex` 中部正文被压成一个巨大 section，结构感不如原 HTML。
   - `HTML Skill 横向对比` 的表格转文字后出现 `审美claude-design 文章article-magazine` 这种挤在一起的感觉。

3. 图片和正文关系还不够像公众号。
   - 首屏图片有视觉，但部分图片更像“网页截图/封面复用”，不是专门为正文服务的公众号插图。
   - 文章内图片前后说明还偏机械，例如“图 1 / 图 2”可以更口语、更有读者引导。

4. HTML 视觉保留有上限。
   - 微信能保住 inline style 的基本层级和色块，但复杂 grid/table/互动会退化。
   - 表格在手机上容易变成长文字堆叠。
   - 大段 CSS 视觉不是越多越好，公众号版要主动改写成阅读流。

5. 发布前验收还缺“读者视角”。
   - 这次验收了标题、字数、图片、封面、发表记录。
   - 但没有在发表前强制做“手机阅读前 3 屏是否舒服”的人工判断。
   - 所以技术上成功，内容体验仍有瑕疵。

## 单篇判断

### 1. 真实浏览器爬取方案重新评定

进步：
- 首屏有明确技术选题和封面图。
- 正文完整度最高。
- 结构信息足，适合工具选型类读者。

问题：
- 表格/矩阵信息太密。
- 第一屏标题重复：微信标题、封面标题、正文 H1 都在说同一件事。
- 需要把“工具列表”改成更像公众号的分段判断，而不是网页报告。

下次改：
- 保留 1 张总览图，后面的工具评分改成 3-5 个结论块。
- H1 不再重复微信标题，改成一句导语。
- 表格只保留最关键 4 列，其他信息写进段落。

### 2. 飞书接入 Codex

进步：
- 首屏最像一篇可读文章。
- 选题清楚，读者一眼知道在讲手机、飞书、本机 Codex 的衔接。
- 封面风格和正文主题一致。

问题：
- 正文中部 section 太长，手机阅读会累。
- 标题也重复。
- 末尾出现本地 HTML 成品路径，不适合公开文章。

下次改：
- 去掉公开文章里的本地路径。
- 每个大段拆成更短的小节，控制单屏内有小标题或引导句。
- 把“检查层/公开边界”做成两张轻量信息图，而不是长表格。

### 3. HTML Skill 横向对比

进步：
- 新版去掉了“测试勿发”。
- 首屏主题明确，封面图比旧版更适合公众号。
- 结论前置，比旧版更像文章。

问题：
- 这是三篇里表格退化最明显的一篇。
- `审美claude-design 文章article-magazine...` 这种表格文本挤压影响阅读。
- 部分代码反引号和英文词堆在正文里，公众号读者会觉得硬。

下次改：
- 表格不要直接粘，改成四张纵向角色卡。
- 每个 skill 用同一模板：一句定位、最适合、不要用来做什么、和谁搭配。
- 英文 skill 名保留，但中文解释要更像读者语言。

## 下一版固定改进规则

1. 公众号发布版不要直接等于 HTML 发布版。
   - HTML 母版保留设计。
   - 公众号版必须再做一层“阅读改写”。

2. 首屏只保留一个主标题。
   - 微信标题已经存在。
   - 封面可以有标题。
   - 正文开头不要再放同一句 H1，改成 1 段导语或一句判断。

3. 表格默认改卡片。
   - 横向表格只适合 HTML 母版。
   - 公众号版用纵向卡片、编号、短结论。

4. 图片不只要“有”，还要承担一个结论。
   - 一张图只讲一个结论。
   - 图前要有铺垫，图后要有解释。
   - 不要让图片只是封面或网页截图。

5. 发布前增加 5 项人工验收。
   - 首屏是否重复标题。
   - 前 3 屏是否有节奏变化。
   - 表格是否在手机上变成堆字。
   - 是否残留本地路径、测试字样、工具内部痕迹。
   - 图片是否服务于正文结论。

## 下次推荐流程

```text
HTML 母版
-> 公众号阅读改写稿
-> 表格转卡片 / 交互转 PNG
-> inline CSS 发布版
-> 固定 Chrome 草稿
-> 手机前三屏验收
-> 保存草稿截图
-> 发表
-> 发表记录 + 公开链接核对
```

## 本次结论

这次最大的进步是“能稳定发出来，而且首屏不再崩”。但真正要变成高质量公众号文章，还差一层平台化编辑：少一点网页搬运，多一点公众号阅读节奏；少一点表格和工具痕迹，多一点结论、卡片和图文引导。
