当前能实现什么

把“已经实现”“可提示注入”“仍需要宿主执行”分开，避免 Grok 把计划说成事实。

模块已实现能力真实边界
Grok CLI / OAuth login、status、refresh、logout、chat、search、image、video、tts、stt、usage、model、state 等命令面保留。 桌面聊天主要通过发布目录内 `grok-cli.exe chat --json --prompt-file`；媒体命令保留入口，UI 深度封装仍有限。
WPF 客户端 三栏 UI、多线程历史、右键重命名、Enter 发送、停止任务、复制、记忆、继续、导出 Markdown。 消息渲染偏朴素，没有富 Markdown/代码高亮，也没有大历史虚拟化。
AgentMemory REST 检索、保存、健康检查、自启动尝试，Viewer 在 `http://localhost:3113`。 图谱、自动压缩、自动注入开关未启用；保存 metadata 还不够丰富。
Skills 扫描 147 个 skill，按任务模式选择相关摘要，最多展开少量完整 skill，避免一次塞爆上下文。 不能自动执行 skill 内部 CLI，相关性还是轻量关键词匹配。
MCP 扫描配置、展示 server、保存启用状态、按模式注入白名单，当前启用 27 个。 UI 已标成 “MCP 白名单（提示注入）”。当前不是完整工具调用 runtime，Grok 不能像 Codex 一样在回合里直接调用所有 MCP。
file-reader 用户明确给路径时，读取白名单内文本文件或目录摘要，并注入 Grok prompt。 白名单：`F:\codex`、`C:\html`、`F:\obsidian\BrainVault`。拒绝 auth/token/secret/env/key 等敏感文件。

风险与改进

这里是给审查者挑刺的靶子。不要为了好看把边界藏起来。

P1 仍需继续改

MCP 不是完整执行层 grok-cli 已随发布版复制 历史仍是 JSON 单文件 文件读取非流式

Windsurf 指出的发布依赖、符号链接、敏感名、原子写、异常兜底和 MCP 文案已经回修；真实 MCP bridge 与 UI 自动化仍是下一阶段重点。

P2 中期优化

Markdown 富渲染 Skills embedding 检索 AgentMemory metadata 统一 SecretRedactor UI 自动化测试

这些更多是体验、可维护性和测试可信度，不应该被包装成已完成。
