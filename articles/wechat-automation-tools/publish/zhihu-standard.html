先说结论：如果任务涉及公众号后台、固定登录态、发表记录、公开页验收，最稳的主流程不是重新开一个浏览器，也不是直接用底层 CDP 拼命令，而是用 Playwright connectOverCDP 接管已经登录的固定 Chrome。

OpenCLI 很适合快速看状态，CDP Bridge 适合临时接管当前标签页，raw CDP 适合排障，playwright-cli 适合公开页探索。但只要进入“要发、要验、要留证据”的正式流程，主线应该收敛到 Playwright connectOverCDP。

一、为什么不能只说“能打开网页”

浏览器自动化最容易误判的地方，是把“能打开页面”当成“能跑生产流程”。公众号后台这类任务真正依赖的是三件事：

第一，登录态要稳定。新开的临时浏览器通常没有后台登录态，看起来干净，实际一进入后台就被扫码、权限、cookie 状态卡住。

第二，任务要能回到公开页验收。后台显示保存成功、草稿正常，不代表读者看到的公开页正常。真正要验的是标题、正文、图片加载、移动端宽度、表格是否变形。

第三，过程要能留下证据。只口头说“发布成功”不够，至少要有公开链接、桌面截图、手机截图和 DOM 统计。

二、这 5 类工具应该怎么分工

我的实测结论是：这些工具不是互相替代关系，而是分工关系。

1. Playwright connectOverCDP：主流程

它的价值在于接管已经登录的真实 Chrome，再用 Playwright 的 API 去做稳定操作。后台读取、发表记录、公开页截图、DOM 检查，都可以脚本化。

如果你要做的是长期流程，而不是临时点两下页面，它应该是主线。

2. OpenCLI：快速检查

OpenCLI 的优势是快。打开页面、看当前 URL、确认页面标题、临时读一点 state，都很顺手。

但它不适合承担复杂批处理。把它放在“快速确认”位置，比让它抢主流程更合适。

3. CDP Bridge：接管当前页

CDP Bridge 适合在桥服务正常时接管当前真实标签页，做临时 JS 执行和截图。它解决的是“我已经在这个页面上了，现在要让自动化接一下手”。

但它不是 Chrome CDP 本身，不能看到浏览器开了调试端口就默认桥可用。桥服务要单独检查。

4. raw CDP：排障工具

raw CDP 最直接，也最啰嗦。它适合确认端口、标签页、Runtime.evaluate、截图链路这些底层问题。

我的建议是：排障时用它，日常发布不要用它写主流程。

5. playwright-cli：公开页探索

playwright-cli 很适合截公开页面、探索选择器、生成脚本思路。但它默认会新开浏览器，不适合依赖后台登录态的任务。

三、最关键的验收动作：从发表记录拿公开链接

这次复盘里，我觉得最值得固化的一条规则是：不要只看后台，不要只看编辑器，要从发表记录里拿公开链接，再回到公开页验收。

[[ZH_WECHAT_AUTO_IMG_2]]

公开页验收至少看这些项目：

  • 标题是否正确；
  • 正文是否完整；
  • 图片数量是否符合预期；
  • 图片是否真正加载；
  • 移动端是否可读；
  • 表格有没有被压扁；
  • 有没有乱码、残缺词、空白图。

这一点和很多内容平台都一样：后台状态只是中间状态，公开页才是最终状态。

四、为什么我最后选 Playwright connectOverCDP 做主流程

因为它同时满足三个条件：复用真实登录态、脚本能力足够强、验收链路完整。

[[ZH_WECHAT_AUTO_IMG_3]]

很多工具只满足其中一项。比如 OpenCLI 很快，但不适合复杂流程；raw CDP 很底层，但维护成本高；playwright-cli 很方便，但登录态不是同一个；CDP Bridge 能接当前页，但还依赖桥服务状态。

所以我的排序不是“谁最强”，而是“谁最适合放在主流程”。正式发布和验收，最怕的不是工具少一步能力，而是每次都换工具、换浏览器、换状态，最后不知道问题出在哪。

五、最终建议

如果你也在做类似的浏览器自动化，我建议按这个顺序来：

  1. 先固定一个长期使用的 Chrome 登录态；
  2. 主流程用 Playwright connectOverCDP；
  3. OpenCLI 只做快速状态确认；
  4. CDP Bridge 只在需要接管当前页时使用；
  5. raw CDP 只做排障；
  6. playwright-cli 用来探索公开页，不负责后台主流程；
  7. 发布后一定从公开页反向验收。

这套方法看起来比“直接开个浏览器跑脚本”麻烦一点，但它的好处是状态清楚、边界清楚、失败也能定位。对内容发布这种一旦出错就很显眼的任务，这比单次跑通更重要。
